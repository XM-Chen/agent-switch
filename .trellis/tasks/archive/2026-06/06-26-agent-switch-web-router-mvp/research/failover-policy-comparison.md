# 故障转移策略对比研究

## 研究问题

为 agent-switch 第一版设计 Claude Code + Codex 本地路由的故障转移策略，需要综合参考：

- `9router`：`ref-9router/master`
- `ccs`（cc-switch）：`ref-cc-switch/main`
- `cpa`（CLIProxyAPI）：`ref-cli-proxy-api/main`
- `sub2api`：`ref-sub2api/main`

重点判断：哪些错误应触发 fallback / retry / failover，哪些错误不应触发，以及第一版是否需要冷却、健康检查、优先级、轮询、粘性会话等机制。

## 四项目对比表

| 项目 | 触发故障转移/重试 | 默认不触发或谨慎触发 | 调度/冷却机制 | 对 agent-switch 的启发 |
|------|-------------------|----------------------|---------------|--------------------------|
| `9router` | 网络错误、超时、429、配额/限流文本、502/503/504、401/403 会尝试刷新 token 或切换、部分 404/402、其他未匹配错误也可能按瞬时错误处理 | 依赖 `ERROR_RULES` 判断；部分错误若规则判定不可回退则直接返回 | 账号级 fallback、指数退避、长/短冷却、优先级、轮询、粘性 round-robin、账户不可用标记 | Codex/账号路由可学习其多账号 fallback、模型前缀路由和 OAuth token 刷新思路，但第一版不宜默认“所有未知错误都 fallback” |
| `ccs` | 网络错误、超时、转发失败、供应商不健康、流式静默超时；除 400/405/406/413/414/415/422/501 外，多数 4xx 和 5xx 可重试；配置/转换/认证错误也可进入 failover 队列 | 400、405、406、413、414、415、422、501；无可用供应商；数据库错误或未知内部错误 | 按供应商队列优先级故障转移，队列顺序可拖拽；健康检查；部分熔断/不健康状态 | Claude Code 路由可学习其“更积极的供应商级 failover”：401/403/404/429 等可能因为换 key、换端点、换模型映射而恢复 |
| `cpa` | 网络/上下文取消/超时；408；500/502/503/504；429 根据 `RESOURCE_EXHAUSTED`、`QUOTA_EXHAUSTED`、`RATE_LIMIT_EXCEEDED` 分级处理；部分 403 可按配置重试 | 400 不重试；401/402/403/404 多数会使凭据冷却或禁用，而不是简单原请求重试；模型不支持会挂起模型状态 | round-robin、fill-first、max-retry-credentials、凭据冷却、配额冷却、可持久化冷却状态、session affinity | OpenAI/Codex 网关可学习“错误分类 + 凭据冷却 + 最大尝试次数”，避免同一坏凭据反复尝试 |
| `sub2api` | 网络/超时；401/403；429；529；>=500；Antigravity 429/503 智能重试；部分服务端配置类 400；PromptTooLong 在特定平台可触发上层 fallback | 普通 400 通常不触发；404/422 在部分平台不触发；流式内容已写出后不再 fallback；Beta Policy Block 不 fallback；余额不足可能永久禁用账号 | 负载感知选择、TopK、粘性会话、粘性逃逸、账号临时不可调度、429/529 冷却、模型级限流 | 请求日志和调度可学习其“摘要记录 + 可解释调度 + 冷却”，但第一版不应引入完整 TopK/粘性会话 |

## 共同结论

四个项目都不是简单地“非 2xx 就换下一个”。成熟项目普遍会区分：

1. **瞬时基础设施错误**：网络错误、连接失败、DNS、超时、408、502、503、504，通常可重试/可切换。
2. **容量/限流错误**：429、529、quota/rate limit/capacity exhausted，通常可切换，并应给失败端点设置冷却。
3. **认证/权限错误**：401/403 是否切换取决于场景：
   - 若上游账号/token 可刷新，应先刷新或标记账号冷却。
   - 若同一路由有多个端点/API Key，切换可能恢复，因此 `ccs` 会更积极地 failover。
   - 若是全局客户端配置错误，则切换无意义。
4. **客户端请求错误**：400、405、406、413、414、415、422、501，多数项目默认不切换。
5. **模型/路径/协议错误**：模型不存在、路径错误、协议不支持，成熟项目有时会根据路由映射场景切换，但第一版应谨慎处理，避免掩盖配置错误。
6. **流式响应已开始输出后**：不应继续 fallback，否则会拼接两个上游的流，导致响应损坏。

## 推荐给 agent-switch 第一版的策略

agent-switch 第一版应采用“默认安全 + 可解释 + 端点冷却”的策略：

### 1. 默认触发 fallback 的条件

以下错误默认触发故障转移，并记录到请求摘要日志的 `fallback_chain`：

- `network_error`：连接失败、DNS 错误、连接被拒绝。
- `timeout`：请求超时、首 token 超时、流式静默超时。
- HTTP `408`。
- HTTP `429`。
- HTTP `500`、`502`、`503`、`504`。
- HTTP `529`。
- 可识别的容量/限流文本：
  - `rate limit`
  - `too many requests`
  - `quota exceeded`
  - `capacity exhausted`
  - `overloaded`
  - `temporarily unavailable`
- 上游连接中断且尚未向客户端写出响应正文。

### 2. 默认不触发 fallback 的条件

以下错误默认不触发故障转移，直接返回给客户端，并在日志中标记为 `non_retryable`：

- HTTP `400`：参数错误、请求体格式错误。
- HTTP `405`、`406`、`413`、`414`、`415`、`422`、`501`。
- 明确的 `invalid_request`。
- 明确的 `context_length_exceeded` / prompt too long。
- 请求正文已经开始向客户端输出后的任意错误。
- agent-switch 本地内部错误：数据库错误、配置解析错误、无可用端点。

### 3. 第一版需要“谨慎切换”的条件

以下错误不应简单归入全局默认，应按路由类型和用户配置处理：

- HTTP `401` / `403`：
  - 对 Codex OAuth 账号：先尝试 token refresh；刷新失败则标记该账号/端点不可用，并尝试下一个可用端点。
  - 对 API Key 端点：如果同一路由中存在多个端点，可尝试下一个端点；同时给当前端点设置较长冷却。
  - 如果所有端点都返回 401/403，应停止重试并提示用户检查凭据。
- HTTP `404` / `model_not_found`：
  - Claude Code 路由参考 `ccs`，如果存在模型映射/备用端点，可尝试下一个端点一次。
  - Codex `/v1` 路由默认不切换，除非路由规则明确声明允许 `model_not_found` fallback。
- 余额不足 / payment required / credit balance：
  - 不应无限重试。
  - 可标记端点为 `quota_or_balance_exhausted`，进入长冷却或禁用状态，再尝试下一个端点。

### 4. 冷却策略

第一版建议实现轻量冷却，而不是完整熔断器：

- `transient`：网络错误、408、5xx → 冷却 30 秒。
- `rate_limit`：429、529、capacity/rate 文本 → 冷却 60 秒，若响应有 `Retry-After` 则优先使用，但设置最大值。
- `auth`：401/403 且刷新失败 → 冷却 5 分钟。
- `quota_or_balance`：余额/配额耗尽 → 冷却 30 分钟或标记需人工处理。

冷却状态应写入 SQLite，应用重启后可以保留或清理。第一版可以先保留当前运行期状态，并在数据库里保存最近错误和 `next_retry_at`。

### 5. 最大尝试次数

为避免一条请求无限扩散：

- 每个路由最多尝试 `max_attempts = min(3, 可用端点数量)`。
- 同一端点同一请求只尝试一次。
- 请求已经开始流式输出后禁止切换。

### 6. 调度策略

第一版默认使用“优先级顺序 + 冷却跳过”：

1. 按路由规则筛选候选端点。
2. 过滤禁用端点和冷却未结束端点。
3. 按优先级升序选择。
4. 失败后尝试下一个候选端点。

第一版暂不默认启用：

- TopK 智能调度。
- 成本/延迟评分。
- 粘性会话。
- 复杂 round-robin。

但数据模型应预留：`priority`、`weight`、`cooldown_until`、`last_error_kind`、`last_success_at`、`last_failure_at`。

### 7. 高级配置是否暴露

第一版 Web UI 建议：

- 默认只暴露“启用自动故障转移”开关。
- 显示内置策略说明。
- 端点级允许设置：优先级、启用/禁用。
- 高级错误码自定义暂不放普通 UI；可以在后续版本或隐藏 JSON 配置中加入。

## 结论

agent-switch 第一版不应采用“非 2xx 全部 fallback”的粗暴策略，也不应一开始照搬 `sub2api` 的 TopK/粘性会话/完整调度平台。

最优解是：

> 学习 `ccs` 的工具级队列和积极供应商切换，学习 `9router` 的账号 fallback 与 OAuth 刷新，学习 `cpa` 的凭据冷却和最大尝试次数，学习 `sub2api` 的错误分类、529/429 处理和可解释日志。第一版落地为“优先级顺序 + 安全错误分类 + 轻量冷却 + 最大尝试次数 + 不记录正文的 fallback 链路日志”。
