# 思考指南

> **用途**：帮助 AI 在编码前主动检查容易遗漏的问题，并指向具体的项目规范。

---

## 为什么需要思考指南？

大多数问题不是因为不会写代码，而是因为没有提前想到：

- 没有检查跨层数据流 → API、任务、Hook、平台适配之间容易不一致。
- 没有检查复用关系 → 同一个约定在多个平台文件中漂移。
- 没有检查边界情况 → 缺少会话标识、缺少任务、无 Git 提交等场景失败。
- 没有把项目约定写入规范 → 未来会话又从零开始猜。

这些指南用于提醒“该想什么”，具体“怎么写”应落到对应的代码规范文件中。

## 快速检查触发器

### 项目约定

- [ ] 是否在新增或修改文档？→ 默认使用中文，详见 [项目约定](./project-conventions.md)。
- [ ] 用户是否提到 `ccs`？→ 按 `cc-switch` / `ref-cc-switch` 理解。
- [ ] 用户是否提到 `cpa`？→ 按 `cli-proxy-api` / `ref-cli-proxy-api` 理解。
- [ ] 是否需要参考外部项目？→ 保持 `origin` 为本项目默认远程，不要切换默认 push/fetch 目标。

### 跨层问题

- [ ] 功能触及 3 个以上层级（任务、脚本、Hook、平台配置、Agent 文档）。
- [ ] 数据格式在层之间发生变化。
- [ ] 多个平台需要消费同一条工作流规则。
- [ ] 不确定逻辑应该放在哪一层。
- [ ] 新增事件类型、JSONL 记录、配置字段、Hook payload 或命令参数。
- [ ] 平台适配代码开始直接读取未验证的原始 payload 字段。

→ 阅读 [跨层思考指南](./cross-layer-thinking-guide.md)。

### 代码复用

- [ ] 正在写与已有代码相似的逻辑。
- [ ] 同一模式重复出现 3 次以上。
- [ ] 正在把新字段同步到多个位置。
- [ ] 正在修改常量或配置。
- [ ] 正在创建新的工具函数。
- [ ] 两个文件用本地类型断言读取同一个未类型化 payload 字段。
- [ ] 多个平台文件维护同一条 Trellis 规则。

→ 阅读 [代码复用思考指南](./code-reuse-thinking-guide.md)。

### 验证 AI 交叉审查结果

- [ ] 审查声称“用户输入可能是恶意的” → 检查真实数据源是内部清单、用户配置还是外部 API。
- [ ] 审查指出“缺少校验” → 判断数据是否来自可信内部源。
- [ ] 审查声称“行为变化” → 阅读代码注释和任务文档，确认是否是有意设计。
- [ ] 审查发现“测试 bug” → 想象删除被测功能后测试是否仍通过；如果仍通过，可能是同义反复测试。

常见 AI 审查误报：

1. **信任边界混淆**：把内部 bundled JSON/manifest 当作外部不可信输入。
2. **忽略设计注释**：把代码注释中写明的意图当作 bug。
3. **变量误读**：没有追踪变量真实定义，例如 Map 的 key 实际是路径而不是名称。

**验证规则**：所有 CRITICAL/WARNING 发现都必须回到真实代码验证后再排序处理。

## 修改前规则（重要）

> 修改任何值之前，先搜索现有用法。

示例：

```bash
# 搜索你准备修改的值
rg "value_to_change"
```

这个习惯能避免“只改了一处，漏了其他平台/Hook/文档”的问题。

---

## 使用方式

1. **编码前**：根据任务类型阅读相关指南。
2. **编码中**：发现重复、跨层、配置或平台适配变化时，回到指南检查。
3. **任务后**：如果学到新的约定或踩到坑，更新对应 spec。

---

**核心原则**：思考指南提醒“要检查什么”，代码规范说明“具体怎么写”。
