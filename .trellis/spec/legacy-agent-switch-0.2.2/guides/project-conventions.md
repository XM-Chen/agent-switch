# 项目约定

> 本文件记录 agent-switch 项目中跨层通用、需要未来 AI 会话持续遵守的项目级约定。

---

## 文档语言

所有项目文档默认使用中文，包括但不限于：

- README、说明文档、设计文档、任务文档；
- `.trellis/spec/` 下的规范文档；
- agent、skill、command 等面向 AI 的提示词文档；
- 其他用于说明项目行为、架构、约定的 Markdown 文档。

例外情况：

- 用户明确要求使用英文；
- 某个工具、生态或上游模板强制要求英文；
- 需要保留外部项目原文以便准确引用。

后续新增或修改文档时，应优先使用中文。如果发现既有文档仍是英文，应在相关任务范围内逐步改为中文，而不是继续扩散英文模板。

---

## 参考项目简称

agent-switch 会参考若干外部项目。用户定义了以下简称：

| 简称 | 指代项目 | Git 远程仓库名 | GitHub 项目 |
|------|----------|----------------|-------------|
| `ccs` | `cc-switch` | `ref-cc-switch` | `farion1231/cc-switch` |
| `cpa` | `cli-proxy-api` | `ref-cli-proxy-api` | `router-for-me/CLIProxyAPI` |

使用规则：

- 用户提到 `ccs` 时，按 `cc-switch` 理解。
- 用户提到 `cpa` 时，按 `cli-proxy-api` 理解。
- 需要查看或对比参考项目时，优先使用对应的 Git 远程仓库名。

示例：

```bash
git fetch ref-cc-switch
git fetch ref-cli-proxy-api
```

## 远程仓库约定

本项目自身远程仓库继续保持为默认远程：

```text
origin -> git@github.com:XM-Chen/agent-switch.git
```

参考项目只作为辅助远程仓库使用，不应替代 `origin`，也不应把本项目默认 push/fetch 目标切换到参考项目。

当前参考远程包括：

```text
ref-9router       -> https://github.com/decolua/9router.git
ref-cc-switch     -> https://github.com/farion1231/cc-switch.git
ref-cli-proxy-api -> https://github.com/router-for-me/CLIProxyAPI.git
ref-sub2api       -> https://github.com/Wei-Shaw/sub2api.git
```

## 需求澄清与质疑规则

AI 不应机械接受所有用户表述。如果用户的需求、架构选择、产品规则或实现细节存在以下情况，应主动指出并澄清：

- 事实错误；
- 和已确认决策矛盾；
- 设计风险过高；
- 范围过大或不合理；
- 表述模糊，不足以直接实现；
- 可能被误解成多个不同意思。

处理方式：

1. 先说明我当前的理解。
2. 指出不清晰、不合理或可能错误的点。
3. 给出推荐解释或推荐方案。
4. 向用户确认是否按该理解继续。

不要为了迎合用户而跳过必要的质疑。规划阶段尤其要主动暴露歧义、错误假设和不合理设计，避免把模糊需求写成错误方案。

## 决策选项参考来源标注规则

在规划阶段，如果 AI 向用户提出问题并给出多个可选答案（例如 A/B/C），每个选项都应说明该方案的参考来源：

- 仿造或主要参考 `9router`；
- 仿造或主要参考 `ccs`；
- 仿造或主要参考 `cpa`；
- 仿造或主要参考 `sub2api`；
- 综合多个参考项目；
- 或者是基于 agent-switch 约束独立推导的方案。

写优缺点时，不应只写抽象取舍，还应说明该方案与参考项目的关系。例如：

```text
A. 端点级真实链路测试（参考 cpa，吸收 sub2api 的摘要日志）
优点：能验证真实 base URL、凭据、模型和协议链路。
缺点：可能消耗 token；这一风险也是 ccs 后来弱化真实 Stream Check 的原因之一。
```

如果尚未查证四个项目的做法，不应假装知道；应先通过本地参考远程、研究文件或 DeepWiki 查询，再给出参考来源。

## 技术栈决策处理规则

技术栈相关问题不应逐项反复询问用户。如果存在不确定点，应参考：

1. 已确认的产品约束和范围边界；
2. `9router`、`ccs`、`cpa`、`sub2api` 四个参考项目的成熟做法；
3. agent-switch 当前已选择的架构方向；
4. 实现复杂度、可维护性和后续扩展需求。

处理方式：

- 产品意图、范围边界、风险承受度仍需向用户确认。
- 技术栈、库选择、目录结构、默认错误格式、验证命令等工程默认项，应由 AI 先做最优决策并写入 `prd.md` / `design.md` / `implement.md`。
- 技术栈决策完成后，集中总结给用户 review，由用户一次性指出不合理点。
- 后续只讨论用户指出的不合理点，不再把每个低层技术默认项都拆成单独问题。

## 错误做法与正确做法

### 错误

```text
用户说“看一下 ccs”，但按一个普通字符串搜索，或者误认为是当前项目模块名。
```

### 正确

```text
用户说“看一下 ccs”，应理解为查看/参考 cc-switch，并优先使用 ref-cc-switch 远程仓库。
```

### 错误

```text
新增 Trellis spec 或项目设计文档时继续使用英文模板。
```

### 正确

```text
新增或修改文档时使用中文；如果必须引用英文上游内容，应说明引用来源并保持必要范围。
```
